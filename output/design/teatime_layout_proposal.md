# Offspace 티타임 뉴스 콘텐츠 레이아웃 디자인 제안서

> 작성일: 2026-04-02  
> 프로젝트: Offspace 셀프 그로스 에이전트  
> 대상: 티스토리 블로그 게시용 AI 동향 티타임 대화 콘텐츠  
> 등장인물: 코부장, 덱과장, 제대리

---

## 문제 정의

| 문제 | 원인 | 영향 |
|------|------|------|
| 뉴스 출처가 글 하단에만 모임 | 대화 흐름 우선 설계 | "진짜야?" 의구심이 즉각 해소 안 됨 |
| 링크를 글 중간에 넣으면 읽기 흐름 방해 | 인라인 링크의 시각적 노이즈 | 몰입 저하, 이탈률 증가 |
| 신뢰도 검증 수단 부재 | 출처 표시 설계 미비 | 콘텐츠 권위 약화 |

---

## 레이아웃 제안 3가지

---

### 제안 1. Split-View 2단 고정 레이아웃 (대표님 아이디어 기반)

**개념:** 화면을 좌우 두 패널로 나눠 왼쪽은 대화, 오른쪽은 해당 뉴스 소스 카드가 고정 표시. 대화의 특정 단락이 뷰포트에 들어오면 오른쪽 소스 카드가 자동으로 하이라이트됨(sticky scroll 연동).

```
┌─────────────────────────┬──────────────────────┐
│  [대화 영역 - 60%]       │  [소스 카드 - 40%]   │
│                          │                      │
│  코부장: "오늘 OpenAI가  │  📰 출처 #1          │
│  또 발표했네요. 이번엔   │  OpenAI 공식 블로그  │
│  o3 모델이래요."         │  2026.04.01          │
│          ↕ 스크롤 연동   │  ★★★★☆ 신뢰도 높음  │
│  덱과장: "맞아요, 저도   │  [기사 보러가기 →]   │
│  봤는데..."              │                      │
│                          │  📰 출처 #2 (비활성) │
└─────────────────────────┴──────────────────────┘
```

**장점:**
- 대화와 출처가 공간적으로 분리되어 각자 독립적으로 읽기 가능
- 스크롤 연동 시 어떤 출처가 지금 대화에 해당하는지 직관적
- 대표님의 원래 아이디어와 가장 일치
- PC 환경에서 최상의 경험 제공

**단점:**
- 티스토리 기본 에디터로는 구현 불가 (HTML 직접 삽입 필요)
- 스크롤 연동은 JavaScript 필요
- 모바일에서 2단이 좁아져 UX 열화 — 반응형 전환 설계 필수
- 뉴스 소스가 5개 이상이면 사이드바가 길어져 비어 보이는 구간 발생

**티스토리 구현 가능성:** ★★★☆☆
- HTML 직접 삽입 모드 + 인라인 JavaScript로 구현 가능
- 티스토리 스킨이 사이드바를 막을 경우 레이아웃 충돌 주의
- Sticky 소스 패널은 `position: sticky` CSS로 JS 없이도 부분 구현 가능

---

### 제안 2. 인라인 소스 토글 카드 (추천 - 구현 난이도 최적)

**개념:** 대화 내용 중 뉴스가 언급될 때 해당 대사 바로 아래(또는 옆)에 접힌 소스 카드를 배치. 클릭하면 펼쳐지며, 기본 상태에서는 최소화되어 흐름을 방해하지 않음.

```
┌──────────────────────────────────────────────────┐
│  코부장: "오늘 OpenAI가 GPT-5 발표했대요. 이번엔  │
│  진짜 멀티모달이 완성됐다고 하더라구요."           │
│                                                    │
│  ┌─ 📎 출처 확인 ──────────────────── [펼치기 ▼] ─┐│
│  │ OpenAI 블로그 · 2026.04.01 · 신뢰도 ●●●●○     ││
│  └────────────────────────────────────────────────┘│
│                                                    │
│  덱과장: "저도 봤는데요, 특히 코딩 벤치마크가..."  │
└──────────────────────────────────────────────────┘

[펼쳐진 상태]
│  ┌─ 📎 출처 확인 ──────────────────── [접기 ▲] ───┐│
│  │ 📰 GPT-5 발표: 새로운 멀티모달 AI의 등장        ││
│  │ OpenAI 공식 블로그                               ││
│  │ 2026년 4월 1일 게시                              ││
│  │ 신뢰도: ●●●●○ (1차 출처, 공식 발표)             ││
│  │ [원문 보러가기 →]                                ││
│  └────────────────────────────────────────────────┘│
```

**장점:**
- 대화 흐름을 전혀 방해하지 않음 (기본 접힌 상태)
- 의심이 생기는 순간 바로 펼쳐서 확인 가능 — 인지 마찰 최소화
- JavaScript `details`/`summary` HTML 태그로 JS 없이 구현 가능
- 모바일 완벽 호환
- 티스토리 HTML 삽입으로 100% 구현 가능

**단점:**
- 모든 출처를 한눈에 볼 수 없음 (글 전체 출처 개요 파악 어려움)
- 적극적인 클릭 행동을 유도해야 하므로 CTA 디자인이 중요
- 대화 생성 자동화 시 소스 카드 삽입 위치 로직 추가 필요

**티스토리 구현 가능성:** ★★★★★
- `<details>` + `<summary>` 태그 + CSS로 완전 구현
- 외부 스크립트 불필요
- 티스토리 모든 스킨에서 동작
- 콘텐츠 자동 생성 파이프라인과 통합 용이

---

### 제안 3. 타임라인 앵커 + 플로팅 소스 패널

**개념:** 대화 왼쪽에 타임라인 바를 두고, 뉴스 언급 시점에 앵커 아이콘(📎)을 배치. 아이콘 클릭 시 화면 우하단에 플로팅 소스 패널이 팝업 형태로 등장.

```
│  │ 코부장: "오늘 OpenAI가..."
│ 📎│ 
│  │ 덱과장: "맞아요, 거기다가..."
│  │
│ 📎│ 제대리: "저는 구글 Gemini 쪽도..."
│  │
│  │ 코부장: "맞아 맞아, 경쟁이..."

                        ┌──────────────────────────┐
                        │ 📰 출처 #2               │
                        │ Google DeepMind 블로그    │
                        │ 2026.03.30                │
                        │ ●●●●● 공식 출처           │
                        │ [원문 →]          [✕ 닫기]│
                        └──────────────────────────┘
```

**장점:**
- 대화 영역이 100% 너비를 활용 — 몰입감 극대화
- 타임라인 바가 "여기에 근거가 있다"는 신뢰 시각화 역할
- 플로팅 패널로 맥락 유지하며 소스 확인 가능

**단점:**
- JavaScript 필수 (플로팅 패널 제어)
- 모바일에서 플로팅 패널이 콘텐츠를 가릴 수 있음
- 구현 복잡도 가장 높음
- 티스토리 스킨에 따라 z-index 충돌 가능성

**티스토리 구현 가능성:** ★★☆☆☆
- JavaScript 허용 스킨에서만 동작
- 티스토리 보안 정책상 일부 JS 제한 가능

## UX 패턴: 뉴스 출처 날짜 + 신뢰도를 자연스럽게 녹이기

### 신뢰도 5단계 시각화 시스템

| 등급 | 기호 | 색상 | 기준 |
|------|------|------|------|
| 5★ | ●●●●● | #2D6A4F (딥그린) | 공식 발표 (회사 블로그, 공식 보도자료) |
| 4★ | ●●●●○ | #40916C (그린) | 주요 미디어 1차 보도 (TechCrunch, VentureBeat 등) |
| 3★ | ●●●○○ | #F4A261 (앰버) | 유력 테크 블로그, 큐레이션 미디어 |
| 2★ | ●●○○○ | #E76F51 (오렌지) | 소셜 미디어, 2차 인용 |
| 1★ | ●○○○○ | #E63946 (레드) | 루머, 미검증 정보 |

### 날짜 표기 패턴

```
"3일 전" → "2026.03.30" 형식으로 호버 시 노출
→ 오늘(2026.04.02) 기준 상대 시간 + 절대 날짜 병기
→ 신선도: 24h 이내 🔴 최신, 3일 이내 🟡 신선, 7일 이상 ⚪ 참고용
```

### 출처 카드 내 포함 정보 (최소화 원칙)

```
[필수] 미디어명 · 날짜 · 신뢰도 등급
[선택] 기사 제목 (펼쳤을 때만)
[선택] 원문 링크 버튼
[제외] 기사 요약 (대화 내용이 이미 요약)
```

### 대화 내 자연스러운 앵커 표기

```
❌ 나쁜 예: "OpenAI가 발표했습니다 [출처1]"  ← 주석 번호 스타일, 학술논문 같은 느낌
✅ 좋은 예: 대사 하단에 출처 카드 자동 배치  ← 흐름 유지
✅ 좋은 예: "📎 이 뉴스 보기" 미니 버튼     ← 명확한 CTA
```

## 모바일 반응형 전략

### 브레이크포인트 설계

```
PC (1024px 이상):
  - 제안 1 참조: 대화 65% + 소스 힌트 바 35% (sticky)
  
태블릿 (768px - 1023px):
  - 대화 100% + 소스 카드 인라인 (제안 2)
  - 소스 카드 너비 100%, 배경색으로 구분
  
모바일 (767px 이하):
  - 대화 100%, 소스 카드 접힌 상태 기본
  - 터치 타겟 최소 44px 보장
  - "📎 출처 1개" 한 줄 요약 → 탭 시 전체 노출
```

### 모바일 최적화 포인트

1. **소스 카드 기본 상태**: 접힌 상태 (1줄 요약만 보임)
2. **터치 영역**: 출처 토글 버튼 최소 44x44px
3. **폰트 크기**: 대화 본문 16px, 소스 카드 내 13px (최소)
4. **여백**: 대화 버블 간격 16px, 소스 카드 상하 8px
5. **색상 대비**: WCAG AA 기준 4.5:1 이상 유지

## 자동화 파이프라인 연동 고려사항

티스토리 자동 게시를 위한 HTML 생성 시 다음 구조 권장:

```html
<!-- 대화 단위 컴포넌트 구조 -->
<div class="dialogue-turn" data-source-id="1">
  <div class="character-bubble ko">
    <span class="character-name">코부장</span>
    <p>대화 내용...</p>
  </div>
  <details class="source-card">
    <summary>📎 출처 확인 · OpenAI 블로그 · 2026.04.01 · ●●●●●</summary>
    <div class="source-detail">...</div>
  </details>
</div>
```

출처가 없는 대화 턴에는 `<details>` 태그를 생략하면 됨.  
LLM이 대화를 생성할 때 `[SOURCE:url,date,trust_level]` 마커를 삽입하면  
후처리 스크립트가 이를 HTML 소스 카드로 변환하는 파이프라인 구성 가능.
